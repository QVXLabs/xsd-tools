# Quickstart: cross-language interop with xsdb-generated bindings

This example shows the point of xsd-tools in one runnable demo: **five programs,
written in four languages, exchange one XML message** — and every program talks
the same wire format because every binding was generated by `xsdb` from the
*same* schema, [`message.xsd`](message.xsd).

A message is produced by a Python program, then relayed through C, Java, and
Python, and finally consumed by C++. Each relay unmarshals the message
with its generated bindings, appends a `<hop>` recording that it was here, and
re-marshals it for the next program. The final program checks that the message
survived the trip intact.

## Pipeline

```
 ep1 ─Python→C─ ep2 ─C→Java─ ep3 ─Java→Python─ ep4 ─Python→C++─ ep5
 (python)       (c)          (java)            (python)         (c++)
 produce        relay        relay             relay            sink+verify
```

Each arrow is a different language boundary: the bytes one program marshals are
unmarshalled by a program in another language, using bindings generated from the
one schema. Endpoints are connected with ordinary stdin/stdout pipes — xsdb
generates the marshalling code; the transport is plain shell glue.

## The message

[`message.xsd`](message.xsd) is a small request envelope (nesting, attributes,
a faceted `priority` of 1..5, an enumerated `kind`, and a repeated `<hop>`):

```xml
<message>
  <header id="msg-001" kind="request">
    <subject>build the bindings</subject>
    <priority>3</priority>          <!-- xs:int restricted to 1..5 -->
  </header>
  <hop seq="1"><endpoint>ep1</endpoint><lang>python</lang></hop>
  <!-- one <hop> added by each endpoint ... -->
</message>
```

The `<header>` is **preserved unchanged** end to end; each endpoint **appends one
`<hop>`**. After five endpoints there are five hops, `seq` 1..5.

## Prerequisites

| Need | Check | Install |
|------|-------|---------|
| `xsdb` built | `ls ../../build/xsdb` | `cmake -S ../.. -B ../../build && cmake --build ../../build --target xsdb` |
| Python 3 | `python3 --version` | system package |
| C11 + C++11 compilers + Expat | `echo '#include <expat.h>' \| c++ -E -xc++ - >/dev/null` | `apt install libexpat1-dev` / `brew install expat` / `dnf install expat-devel` |
| JDK 17 | `javac -version` | `apt install openjdk-17-jdk` / `brew install openjdk@17` |

> The example is built for **JDK 17** (`--release 17`); the *generated* bindings
> themselves stay JDK 8 compatible, so consumers of `java-xml-stax` output are
> not forced onto a newer JDK.

## Reproduce

### A. One command, via CMake (recommended)

From the repository root:

```sh
cmake -S . -B build -DQUICKSTART=ON
cmake --build build --target run-quickstart
```

This generates the four bindings with `xsdb`, compiles the C, C++, and Java
endpoints, and runs the relay, printing the message at every stage.

### B. By hand (to see exactly what the target does)

```sh
cd examples/quickstart
XSDB=../../build/xsdb
export XSDTOOLS_DATA=../../templates       # so xsdb finds the templates
mkdir -p gen/c gen/cpp gen/java gen/javac

# 1. generate the bindings — one xsdb invocation per language
"$XSDB" python-sax    message.xsd > gen/message.py
"$XSDB" --out-dir gen/c    c-xml-expat    message.xsd
"$XSDB" --out-dir gen/cpp  cpp-xml-expat  message.xsd
"$XSDB" --out-dir gen/java java-xml-stax  message.xsd -package interop

# 2. compile the native endpoints (Expat for C/C++, JDK for Java)
cc  -std=c11   -Igen/c   -I. $(pkg-config --cflags expat) \
    ep2_relay.c   gen/c/xml_message.c     $(pkg-config --libs expat) -o gen/ep2
c++ -std=c++11 -Igen/cpp -I. $(pkg-config --cflags expat) \
    ep5_sink.cpp  gen/cpp/xml_message.cpp $(pkg-config --libs expat) -o gen/ep5
javac --release 17 -d gen/javac gen/java/*.java Ep3Relay.java

# 3. run the relay (each endpoint prints what it received, then forwards it)
./run.sh gen
```

## Expected output

`run.sh` prints each endpoint's output (what the next one receives); the message
grows by one `<hop>` per stage. Abridged (long lines wrapped):

```
──────── ep1 (python) — produce the message ────────
<?xml version="1.0" encoding="UTF-8" ?>
<message><header id="msg-001" kind="request"><subject>build the bindings</subject>
<priority>3</priority></header><hop seq="1"><endpoint>ep1</endpoint><lang>python</lang></hop></message>

──────── ep2 (c) — relay ────────
…<hop seq="1">…</hop><hop seq="2"><endpoint>ep2</endpoint><lang>c</lang></hop></message>

──────── ep3 (java) — relay ────────
…<hop seq="3"><endpoint>ep3</endpoint><lang>java</lang></hop></message>

──────── ep4 (python) — relay ────────
…<hop seq="4"><endpoint>ep4</endpoint><lang>python</lang></hop></message>

──────── ep5 (c++) — sink + verify ────────
PASS: message relayed through 5 endpoints (python->c->java->python->c++); header intact, 5 hops in order.
<?xml version="1.0" encoding="UTF-8"?><message><header id="msg-001" kind="request">
<subject>build the bindings</subject><priority>3</priority></header>
<hop seq="1"><endpoint>ep1</endpoint><lang>python</lang></hop>
<hop seq="2"><endpoint>ep2</endpoint><lang>c</lang></hop>
<hop seq="3"><endpoint>ep3</endpoint><lang>java</lang></hop>
<hop seq="4"><endpoint>ep4</endpoint><lang>python</lang></hop>
<hop seq="5"><endpoint>ep5</endpoint><lang>c++</lang></hop></message>
```

The `PASS` line is written to stderr; `ep5` exits 0 on success, non-zero
otherwise.

## How it works

- **`ep1_producer.py`** — builds a `message` object (header + first hop) with the
  generated `python-sax` classes and marshals it to stdout. No input.
- **`ep5_sink.cpp`** — `cpp-xml-expat` (C++ streaming). The generated
  `Marshaller` reports each element through an `on_<name>` hook at its *end* tag
  (post-order), so [`relay_util.hpp`](relay_util.hpp) captures the typed elements
  and re-emits them in schema order with one extra `<hop>`. `ep5` also asserts
  the header survived and five ordered hops accumulated.
- **`ep2_relay.c`** — `c-xml-expat` (C streaming). The same idea in C: the
  generated `xml_marshaller` reports each element through `unmarshal_<name>`
  *function pointers* (post-order); the endpoint copies out the values it needs,
  then re-marshals through the nested `xml_marshal(...)` function-pointer API,
  appending its `<hop>`.
- **`Ep3Relay.java`** — `java-xml-stax`. `Document.unmarshal(xml)` returns the
  tree; the new hop is `m.children().add(hop)`; `Document.marshal(m)` writes it
  back.
- **`ep4_relay.py`** — like ep1 but reads the message from stdin first.

Every endpoint only fills in or reads values — all the parsing and serialization
code is generated by `xsdb`.

## Try it

- **Change the subject** in `ep1_producer.py` (`subject._content = [...]`) and
  re-run. It rides through all five endpoints unchanged and `ep5` still prints
  `PASS` — the subject is carried but not asserted. (`ep5` *does* check the rest:
  exactly five hops in order, `kind="request"`, and `priority == 3`, so changing
  those will trip its assertion — that's the verifier doing its job.)
- **Break a facet:** set `priority._content = [9]` in `ep1_producer.py` (the
  schema restricts it to 1..5). The *next* endpoint — `ep2`, in C — rejects the
  out-of-range value while unmarshalling (`priority: facet validation failed`,
  then aborts with a non-zero exit): facet validation enforced *across the wire*,
  in a different language than the producer.
